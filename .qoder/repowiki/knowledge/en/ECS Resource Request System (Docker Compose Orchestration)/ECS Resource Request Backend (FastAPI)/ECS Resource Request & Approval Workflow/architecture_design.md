Two FastAPI `APIRouter` modules expose the workflow: `routers/requests.py` (user-facing `/requests`, guarded by `get_current_user`) handles listing a user's own requests, creating new ones with an active-request cap (`MAX_ACTIVE_REQUESTS_PER_USER`), and initiating deletion of approved resources; `routers/approvals.py` (admin-only via router-level `dependencies=[Depends(require_admin)]`) exposes pending queues, region info, VPC/vswitch/security-group/DNS lookups against Alibaba Cloud, and batch approve/reject/deletion-decision endpoints. All state mutations are delegated to `services/approval.py`, which implements per-request transactional loops (`db.commit()` after each item) so partial-batch failures do not roll back earlier successes. The service layer owns instance-name generation (`_next_instance_name` based on sanitized user name + per-user sequence), password encryption via `app.services.crypto`, audit logging through `AuditLog`, and calls into `aliyun_ecs` / `aliyun_vpc` / `aliyun_dns` for cloud operations. Dependency direction is strictly routers → services → external SDKs; models/schemas are imported but never mutated directly from routes.
